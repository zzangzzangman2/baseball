# Codex 설계도: 게임캐스트 v2 클린룸 재구축 — "이미지가 기하다"

작성일: 2026-07-07
결정: 현 게임캐스트 렌더 계층은 수정 불가 수준으로 꼬였다(아래 §1). **렌더/아트/안무만 처음부터 재구축**한다. 엔진(PA 이벤트)·페이싱(배속/홀드/스텝/일시정지)·HUD DOM·랩/검증 인프라는 **그대로 재사용**한다.

---

## 1. 왜 꼬였나 — 재구축에서 반복 금지할 교훈 4개

1. **코드가 기하를 그렸다.** 필드를 절차 드로잉으로 그리다 보니 외야(원근)·내야(무원근)·스프라이트(단일 크기)의 좌표계 3개가 충돌했고, 손으로 찍은 좌표 상수가 수십 곳에 흩어져 무엇을 고쳐도 계속 어긋났다. → **v2에서 코드는 기하를 그리지 않는다. 기하는 이미지에 굽고, 코드는 앵커에 스프라이트를 얹기만 한다.**
2. **안무가 progress-if-else 덩어리였다.** 타석 연출이 수백 줄 조건문이라 "주자가 3루로 달리는" 버그를 찾기도 어려웠다. → 선언적 타임라인 컴파일러로.
3. **렌더러가 2개였다.** Phaser+Canvas 폴백 이중 유지보수가 모든 수리를 2배로 만들었다. → **Phaser 단일**(폴백은 "정지 필드+텍스트 피드"만).
4. **눈검수가 늦었다.** 좌표 어긋남은 코드리뷰로 안 보인다. → 앵커 디버그 오버레이 + 버스트 캡처를 개발 루프 안에.

## 2. 새 아키텍처 (모듈 경계)

```
src/gamecast2/
  assets 로더  ← assets/gamecast2/field-*.png + field-*.anchors.json + players-*.png(아틀라스)
  timeline.js  ← compilePlayTimeline(event, anchors) → 트랙(키프레임) : 순수함수, 단위테스트 대상
  scene.js     ← Phaser 씬: 배경이미지 1장 + 스프라이트 풀(y-sort) + 공 + FX + 카메라
  index.js     ← mountGamecast2({screen, sequence, playbackStore...}) : 기존 컨트롤러 계약 유지
```
- 기존 `state.ui.gamecastEngine`에 `"v2"` 추가 — 랩 `?engine=v2`로 병행 개발, 체크리스트 통과 후 기본값 전환, 2주 뒤 구 렌더러 삭제.
- 페이싱(P 시리즈)·playbackStore·HUD DOM(칩/핍/점보)·사운드는 인터페이스 그대로 물려받는다(재작성 금지).

## 3. 핵심 발명: 필드 = 이미지젠 에셋 + 앵커맵

### 3.1 필드 마스터 이미지 (Codex가 이미지 생성)
- **한 장의 완성된 구장 그림**에 모든 기하를 굽는다: 관중석(원근), 담장+워닝트랙, 파울폴(담장 코너에), 잔디 모잉 스트라이프, 흙 내야(베이스패스 띠+잔디 컷+원형 마운드+홈 흙원), 배터박스+홈플레이트, 파울라인(홈→폴에서 종료), 전광판(빈 패널로 — 코드가 채움), 더그아웃.
- 규격: **960×720 픽셀아트**(디자인 해상도 480×360의 2배로 생성 후 정수 축소 — 픽셀 크리스프 확보), 팔레트 ≤32색, 안티에일리어싱 금지, 광원 상단 고정.
- 프롬프트 골자(스펙 문서에 템플릿 수록): "top-down 3/4 broadcast view behind home plate, retro pixel art baseball stadium, night game, clean flat colors, no players on field, no text" + 금지어(선수/공/글자/로고).
- 현재 활성 필드는 `field-jamsil-day` 한 장으로 고정한다. `field-jamsil-night`, `field-gocheok-dome` 에셋은 후속 확장용으로만 보존하고 런타임·QA에서는 비활성화한다.
- 후처리 파이프라인 `tools/build_gamecast_field.py`: 팔레트 양자화(기존 스프라이트 빌더 방식), 해상도 검증, 지정 마커색 픽셀 제거.

### 3.2 앵커맵 (위치 문제의 종결)
- 이미지 생성 직후 **마커 픽셀**(예약색 `#ff00ff` 계열 15종)을 편집 단계에서 찍는다: home, 1B, 2B, 3B, mound, 수비 9포지션(P/C/1B/2B/3B/SS/LF/CF/RF), 좌/우 폴 밑동, 전광판 좌상단.
- 빌드 스크립트가 마커를 스캔해 `field-*.anchors.json` 생성 후 마커 픽셀을 주변색으로 지움: `{ home:{x,y,scale:1.0}, second:{x,y,scale:0.86}, cf:{x,y,scale:0.72}, ... }` — **scale = 그 지점의 스프라이트 배율**(깊이감. 3~4단 계단값).
- 경로는 앵커 보간으로만: 주자 = home→1B→2B→3B→home 베지어(이미지의 베이스패스 위), 타구 = home→낙하점(존별 앵커 가중), 송구 = 수비앵커→베이스앵커 직선. **손좌표 상수 전면 금지.**
- **디버그 오버레이 `?debug=anchors`**: 앵커 점+이름+주루 경로선을 화면에 표시 — 위치 어긋남이 1초 만에 보인다. 개발 필수 도구.

### 3.3 왜 이 방식인가
이미지젠은 "아름다운 구장 한 장"을 아주 잘 그린다(절차 코드로는 수십 시간). 코드가 못 하는 건 그림이고, 이미지가 못 하는 건 움직임이다 — 각자 잘하는 것만 맡긴다. 구장별 스킨도 "새 이미지 1장+마커"로 끝난다.

## 4. 스프라이트 v2 (이미지젠 재생성 — 규격 고정)

- **64×64px** 셀(48→상향: 필드가 2배 해상도라 여유 생김), 발바닥 baseline y=60, 중심 x=32. 팔레트/예약색 규칙은 기존 `docs/gamecast-sprite-art-spec.md` 승계(모자 `#d23b3b` 팀컬러 스왑 등).
- 애니메이션 표(기존 아트 지시서 승계): 타격 8(스탠스→로드→스트라이드→스윙1→**컨택(홀드)**→스윙2→팔로우1→2) ×스탠스 3종, 투구 7, 달리기 4사이클, 송구 3, 포구 3, 다이브 3, 슬라이드 2, 포수 2, idle/walk/take/miss. 홈/원정 2벌.
- 시트: 8열 그리드, 셀당 256px로 생성 → 64px 정규화(기존 빌더 확장). **레지스트레이션 자동 검사**(bbox 하단 60±1 아니면 빌드 실패) + frames/durations 길이 검증.
- 심판/코치 1열 추가(회색 팔레트).

## 5. 안무 = 타임라인 컴파일러 (조건문 금지)

```js
compilePlayTimeline(event, anchors) → {
  durationMs,
  tracks: {
    pitcher:  [{t:0, anim:"pitch", at:"mound"}],
    ball:     [{t:0.10→0.22, path:"mound→home"}, {t:0.30→0.85, path:"home→cf", arc:1}],
    batter:   [{t:0, anim:"swing", at:"home"}, {t:0.30, anim:"run", path:"home→first"}],
    fielders: [{who:"cf", t:0.30, anim:"run", path:"cf→landing"}, {t:*, anim:"catch"}, {t:*, anim:"throw"}],
    runners:  [...basesBefore→basesAfter 경로],
    camera:   [{t:0.30→0.9, follow:"ball", zoom:1.10}],   // 홈런/장타만
    captions: [{t:0.86, badge:"홈런!"}],                    // 항상 연기가 끝난 뒤
    sfx:      [{t:0.30, id:"crack"}]
  }
}
```
- 아웃컴별 템플릿 12종(삼진/볼넷/단타/2·3루타/홈런/내야아웃/외야아웃/병살/실책/도루/폭투없음→생략/이닝종료)을 **데이터**로. scene.js는 트랙 재생만.
- **단위 테스트**(verify에 추가): 주자 경로가 반드시 home→first 방향(x 증가 방향) / 공 낙하점=수비수 도착점 / 배지 시점 > 주루 종료 시점 / 모든 anim 키가 아틀라스에 존재.
- 배속 불변식(기존 PF-2): ball 트랙 구간은 최소 실시간 보장, 나머지 압축.

## 6. 장면 연출 기준 (고퀄의 실체)

1. **투구 루틴**: 와인드업→릴리스→공 궤적(0.35s+)→미트/컨택. 삼진·볼넷은 2~3구 요약.
2. **깊이감**: 앵커 scale로 원거리 축소 + 어두운 배경 위 공은 밝은 트레일+1px 글로우.
3. **카메라**: 기본 고정. 장타/홈런만 공 팔로우 팬(≤12%, easeInOut), 득점 홈인 클로즈. reduced-motion 오프.
4. **여운**: 홈런 = 담장 통과 플래시→관중 점멸→주자 라운딩 완주→그때 점수/배지. 이닝 교대 = 수비 조깅 등퇴장.
5. 전광판은 코드가 채운다(이닝/스코어/타자명 블록레터) — 빈 패널 금지.

## 7. 마이그레이션 마일스톤 (각각 커밋+체크)

| 단계 | 내용 | 통과 기준 (버스트 캡처 + ?debug=anchors) |
| --- | --- | --- |
| R1 | 잠실 주간 필드+앵커맵+빌더+디버그 오버레이 | 잠실 canonical 앵커가 그림의 올바른 위치에 찍힘 |
| R2 | scene.js: 배경+수비 9명 배치+아이들 | 수비가 전원 제 포지션, 깊이 스케일 적용 |
| R3 | 타임라인 컴파일러+투구/삼진/볼넷 | 투구 루틴 항상 보임, 단위테스트 PASS |
| R4 | 타구/안타/아웃/병살/홈런+주루 | 주자 1루 방향, 수비 이동→포구→송구 연기, 배지 후행 |
| R5 | 카메라+사운드 연결+잠실 주간 스킨 고정 | 홈런 팔로우, 모든 선택 경로가 잠실 주간으로 귀결 |
| R6 | 페이싱 통합(배속/홀드/스텝)+verify 통합, 기본 엔진 전환 | 기존 verify:browser 랩 검사 전부 PASS |
| R7 | 구 렌더러 삭제(2주 후) | — |

## 8. 가드레일

1. 엔진/경기 결과/페이싱 로직 무변경. `verify` 3종 상시 PASS.
2. 이미지젠 에셋도 결정론 원칙: 생성물은 리포에 커밋(재생성 금지), 빌더는 순수 변환.
3. 실명/로고 금지(이미지에 글자·구단 로고 넣지 말 것 — 전광판 텍스트는 코드 렌더).
4. 스프라이트/필드 규격 변경 시 스펙 문서 동시 갱신. 마커색은 게임 팔레트와 절대 겹치지 않게.
5. 개발 루프: 매 R단계마다 `node tools/capture_gamecast_shots.mjs burst` + `?debug=anchors` 스크린샷을 커밋 메시지에 체크리스트로 남길 것.
