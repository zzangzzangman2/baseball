## 1. 문제 진단: 왜 "지나가버리는가"

수치로 보면:
- watch 모드 타석 슬롯 = **2600ms + 갭 340ms ≈ 2.9초**(ui.js:144-145). 투구→타격→주루→결과가 그 안에 압축.
- 배속 옵션이 **[x1, x2, x3, x4] — 올리는 것만 있다**(ui.js:148). 내리는 것(x0.5)도, **일시정지도, 타석 단위 정지도 없다**. 스킵은 경기 끝 일괄 점프뿐.
- 즉 유저가 상황을 읽고 반응할 수 있는 최소 단위(멈춤)가 존재하지 않는다. "개입하려고 하면 지나간다"는 당연한 결과.

그리고 정직하게 짚을 것: **현재 구조(선시뮬 리플레이)에서 '개입'은 원천적으로 연출일 수밖에 없다** — 결과가 이미 계산돼 있다. P 시리즈는 "따라가며 볼 수 있게" 만드는 것이고, 결과를 바꾸는 진짜 개입은 Stage B(인터리브 엔진)+Stage C(감독 모드)다. §4에서 착수를 권고한다. P-3의 홀드 UI는 Stage C의 결정 창으로 그대로 승격되도록 설계해 둔다.

## 2. P 시리즈 — 페이싱과 컨트롤 (이번 사이클 최우선)

### P-1. 일시정지 (즉시)
- ⏸/▶ 버튼(속도 버튼 옆) + **Space 키** + **캔버스 클릭 토글**. Phaser `pause()/resume()`은 이미 있고(가시성 처리에 사용 중) UI 배선만 없다.
- 일시정지 중: 화면 어둡게+"일시정지" 오버레이, 현재 상황 카드(이닝/아웃/주자/스코어/타석 매치업) 표시. 재개 시 0.4s 카운트 후 진행.
- playbackStore에 paused 저장 — 재렌더에도 일시정지 유지.

### P-2. 배속 재설계 + 스텝 모드
- 속도 옵션을 `[0.5, 1, 1.5, 2, 4]`로 교체(x0.5 = 타석당 ~5.2초). 기본 x1 유지, 선택은 store에 저장되어 다음 관전에도 유지.
- **스텝 모드(타석 확인 모드)** 토글: ON이면 각 타석 재생이 끝나는 순간 자동 홀드 — 결과 카드(타구/결과/스코어 변화)와 "다음 타석 ▶"(Space) 대기. FM의 key-moment 방식.
  - 구현: 슬롯 경계에서 `elapsedMs`를 클램프하고 paused 진입. 다음 입력 시 다음 슬롯부터 재개.
  - `state.ui.gamecastStepMode` 저장. 스텝 모드에서 배속 버튼은 재생 구간 속도만 제어.
- 수용 기준: 스텝 모드 ON에서 사용자 입력 없이 다음 타석으로 넘어가는 일 0회. x0.5에서 타석 전 과정이 읽힘.

### P-3. 자동 홀드 = 개입 창 (핵심 체감)
게임이 스스로 중요한 순간에 멈춰 유저를 기다리게 한다. 전부 토글(기본 ON, `state.settings.gamecastHolds`):
1. **이닝 교대 홀드**: 하프이닝 종료 슬레이트에서 정지 — 이닝 요약(득점/잔루/투구수)+다음 이닝 선두 예고, "계속 ▶". ※ 이 UI 프레임이 Stage C에서 "투수 교체/대타 결정 창"으로 승격된다 — 레이아웃에 결정 버튼 자리를 비워 둘 것.
2. **승부처 홀드**: `gamecastLeverageScore ≥ 0.55` 타석 직전 pre-pitch에서 정지 — "승부처" 배지 + 상황 카드(주자/아웃/점수차/타자 시즌 성적 vs 투수). "계속 ▶"(Space).
3. **득점 직후 여운**: 득점 확정 시 1.2s 슬로우(배속 무시) 후 원래 속도 복귀. 홀드 아님.
- 기존 Q-6 리버리지 홀드(0.3초 리매핑)는 2번으로 흡수하고 제거.
- 수용 기준: 9회 동점 만루에서 자동 정지+배지. 이닝 교대마다 정지(토글 OFF 시 통과). 홀드 중 hidden→복귀에도 홀드 유지.

### P-4. pre-pitch 매치업 그래픽
- 각 타석 시작 0.9s 동안 방송풍 매치업 하단 카드: 타자(시즌 AVG/HR/타점, 좌우) vs 투수(ERA/K, 투). 데이터는 `seasonStats`에서(있는 것만, 추정치 생성 금지).
- 스텝/홀드 중에는 상세 버전. x2 이상 배속에서는 생략(속도감 유지).
- 이 카드가 "읽을 시간"을 제공해 체감 속도를 낮춘다 — 슬롯을 늘리지 않고 정보 밀도로 해결.

### P-5. 타임라인 + 이닝 점프 (Q-7 잔여)
- 진행 바: 지나간 하프이닝 마커(득점 이닝 강조), 클릭 시 그 시점부터 재생(elapsedMs 세팅). 미래 구간은 미표시(스포일러 방지). "다음 득점 장면으로" 버튼.

## 4. Stage B 착수 권고 (경기 진행 방식)

유저가 "개입하고 싶다"는 수요를 명확히 표현했다. 선시뮬 리플레이에서는 P 시리즈가 상한선이다 — **다음 엔진 사이클에서 Stage B(하프이닝 인터리브 루프)를 착수할 것을 권고한다.** 상세 스펙은 `docs/codex-gamecast-v2-work-order.md` §2 Stage B 그대로: WP-07(연장 12회/무승부)과 한 세트, 끝내기/9회말 스킵 지원, 시드 하프이닝 파생, verify:balance 재튜닝 포함. Stage B가 끝나면 P-3의 이닝 교대 홀드에 결정 버튼(투수 교체/대타 예약)만 얹으면 Stage C(감독 모드)다.

## 5. 가드레일 + 검증

1. 경기 결과 로직 무변경(P/R/A 전부 연출·UI 계층). `verify` 3종 PASS 유지.
2. 모든 홀드/카드/티커는 `prefersReducedMotion` 존중(정지 화면으로 대체), deterministic(hashParts).
3. playbackStore 확장 필드(paused/stepMode/speed)는 재렌더·모달 전환에도 보존. 저장(JSON)에는 넣지 않는다(세션 연출 상태).
4. 스텝 모드/홀드는 verify:browser 랩 검사에 추가: 스텝 ON → 3초 대기 → 피드 증가 0건 확인, Space 입력 → 1건 증가.
5. `gamecast-lab.html` 옵션 추가: `?step=1`, `?speed=0.5` (테스트 진입 단축).

```powershell
npm run verify && npm run verify:balance && npm run verify:browser
# 수동: gamecast-lab.html?fps=1&step=1 — 스텝/홀드/일시정지/사운드 확인
```
