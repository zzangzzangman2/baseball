## 0. 정직한 상한선과 전략

현재 아트: 48px 치비 스프라이트 **포즈당 1프레임**(총 18포즈), 소스는 이미지젠 시트(1536×1024) → `tools/build_gamecast_sprites.py`가 팔레트 정규화. 구장은 100% 코드 절차 드로잉 단일 그림.

가능한 최대치를 3등급으로 나눈다:

| 등급 | 무엇 | 누가 | 체감 |
| --- | --- | --- | --- |
| ① 코드만으로 | 수비 안무(루트/다이브/병살), 구장 데이터화·스킨·조명, 관중 v2, 애니 플레이어 | **Codex 단독** | 매우 큼 — "자연스러움"의 절반은 안무다 |
| ② 이미지젠 소스 재생성 | 스윙 8프레임·투구 7프레임 등 **프레임 수 확장**, 타격폼 배리에이션 | **유저(이미지젠) + Codex(파이프라인)** | 가장 큼 — 1프레임→다프레임이 도트 퀄의 본체 |
| ③ 불가/후순위 | 전문 도트 아티스트 손맛(픽셀 클러스터 미세 셰이딩), 초고해상도 도트 | — | 목표 아님. 목표는 "옛 파워프로 2D/베이스볼 슈퍼스타즈급"(기존 스펙 문서와 동일) |

**핵심 판단: 해상도는 올리지 않는다.** 디자인 해상도 400×360, 스프라이트 48px 유지. 이유: ① 정수 배율 확보(픽셀아트 crisp의 전제), ② 지금 어색함의 원인은 픽셀 수가 아니라 **중간 프레임 부재와 수비수가 순간이동하는 안무**, ③ 클로즈업 디테일은 Q-4 카메라 줌이 해결(줌하면 픽셀이 커져도 roundPixels로 crisp). 48→64px 상향은 ②등급 소스 퀄리티가 검증된 뒤 2차 결정.

## 1. Track S — 선수 스프라이트 v2 (역동적인 타격폼)

### S-1. 애니메이션 프레임 확장 (최우선 — 이게 본체)

포즈당 1프레임 → 액션당 프레임 시퀀스. 목표 프레임표:

| 액션 | 현재 | 목표 | 구성 |
| --- | --- | --- | --- |
| 타격 | 3 (stance/swing/follow) | **8** | stance → load(테이크백) → stride(발 들기) → swing1(배트 수평 전) → contact(임팩트, 1프레임 홀드용) → swing2(배트 통과) → follow1 → follow2(원핸드 피니시) |
| 투구 | 2 (windup/pitch) | **7** | set → kick(레그킥 정점) → stride → cock(팔 젖힘) → release → follow1 → follow2 |
| 달리기 | 2 | **4** | 접지A → 공중A → 접지B → 공중B (컨택트-패싱 사이클) |
| 송구 | 0 (없음) | **3** | plant(디딤) → throw(릴리스) → follow |
| 포구 | 1 (catch) | **3** | track(글러브 준비) → reach → squeeze |
| 다이빙 | 1 | **3** | launch → 수평 슬라이드 → getup |
| 슬라이딩 | 1 | **2** | 진입 → 접지(흙먼지는 엔진 파티클 유지) |
| 포수 | 1 | **2** | 프레이밍 → 블로킹 |
| 기타 유지 | idle/walk/miss/take/lookUp | 동일 | 재활용 |

- 시트 그리드: 48px 셀 **8열 × 6행**(여유 포함), 소스 이미지젠 권장 2048×1536(셀당 256px로 생성 후 축소 정규화 — 지금 워크플로와 동일 원리).
- 레지스트레이션 규칙 유지(발 y=45, 중심 x=24) — **빌드 스크립트에 자동 검사 추가**: 각 프레임 불투명 bbox 하단이 45±1 벗어나면 빌드 실패(덜덜 떨림 방지, 스펙 문서의 "가장 중요" 항목을 기계 검증으로).
- `build_gamecast_sprites.py` v2: `POSE_GRID` → `ANIMATIONS = { swing: { frames: [...], durations: [...] } }` 메타를 JSON으로 출력. 구 포즈 키는 첫 프레임 별칭으로 하위호환.
- 렌더러(Phaser+Canvas 공통): pose 스위치 → **애니메이션 플레이어**(이벤트 progress → 누적 duration으로 프레임 결정). contact 프레임은 임팩트 순간 2배 홀드(타격감).

### S-2. 타격폼/투구폼 배리에이션 ("선수마다 다르게")

- 스탠스 3종: 오픈 / 스퀘어 / 크라우치. 소스 시트에 stance~load 구간만 3벌(스윙 이후 프레임은 공유 — 시트 +6프레임으로 절약).
- 배정: `hashParts(playerId)` 고정 + 능력치 보정 — 파워형(power>contact)은 레그킥 큰 load, 컨택형은 짧은 스트라이드. 좌타는 기존 flipX.
- 투구폼 2종: 와인드업 / 쓰리쿼터(사이드암 느낌) — 투수 id 해시 배정.
- 수용 기준: 라인업 9명 관전 시 최소 2가지 폼이 눈에 보임. 같은 선수는 항상 같은 폼(결정론).

### S-3. 외형 개성 (후순위, 소스 재생성에 무료로 얹기)

- 피부톤 2단(예약색 2벌 — 빌드 시 팔레트 스왑), 배팅장갑/손목밴드 포인트색(팀 액센트 재활용), 등번호(이미 구현됨) 유지.
- 체형 변화(슬림/불키)는 프레임 2벌이 필요하므로 **하지 않는다** — 개성은 폼(S-2)과 팔레트로.

### S-4. 소스 시트 재생성 키트 (Codex가 준비 → 유저가 생성)

- `docs/gamecast-sprite-art-spec.md`를 v2로 갱신: 8×6 그리드 좌표표, 프레임별 포즈 설명(위 표), 예약색 규칙(기존 유지), 이미지젠 프롬프트 템플릿(치비 비율/1px 아웃라인/플랫 팔레트/투명 배경 지시 포함).
- **QA 자동화**: `tools/preview_gamecast_anims.py` — 아틀라스에서 액션별 애니 GIF + 컨택트 시트를 `reports/motion-samples/`로 출력. 유저가 시트를 넣고 빌드하면 GIF로 즉시 검수.
- ⚠️ 이 트랙은 **유저가 이미지젠으로 소스 시트를 다시 뽑아야 완성**된다. Codex는 파이프라인·스펙·검증을 먼저 완비하고, 소스가 올 때까지 기존 18프레임에 대해 애니 플레이어가 하위호환으로 돌아가야 한다.

## 2. Track D — 수비 안무 ("자연스럽게"의 절반은 여기)

현재: 수비수 8명이 **고정 위치에 서 있다가** 담당 수비수만 progress 0.82에서 catch 포즈로 바뀐다(ui.js `drawPixelFielders`/`buildGamecastDefenseSprites`). 공만 날아다니고 사람이 안 움직이니 부자연스러운 것.

- **D-1. 타구 추적 루트**: 담당 수비수가 스테이션에서 낙구 지점까지 ease 이동(도착 타이밍 = 공 도착과 동기). 땅볼=차지(전진), 플라이=드리프트/백트래킹. 포구 후 plant→throw(S-1 송구 프레임)→1루 송구 라인, 1루수 스트레치 포구.
- **D-2. 상황별 포즈 선택** (deterministic, 이벤트 데이터 기반): 낙구점이 수비 범위 경계면 다이브(내야)/러닝캐치(외야), 담장 근처 깊은 플라이는 백트래킹+점프캐치 시도(홈런은 담장 위 lookUp — 기존 포즈 재활용), 평범한 타구는 기본 포구. `battedBallType`+`fieldingPosition`+deterministicRange로 선택.
- **D-3. 병살 안무**: 병살 이벤트(doublePlay)는 6-4-3/4-6-3 시퀀스 — 포구→2루 커버 야수 베이스 터치+피벗 점프 스로→1루 스트레치. 주자 2명 동시 이동(선행 주자 2루 앞 아웃 연출).
- **D-4. 주루 디테일**: 베이스 라운딩을 직선→베지어 호로(장타 시 크게 돌기), 슬라이딩은 클로즈 플레이(아웃 or 득점 임박)에만, 득점 홈인 후 더그아웃 방향 하이파이브 1프레임(P-3 득점 슬로우와 결합).
- 수용 기준: 랩에서 좌익수 앞 안타 관전 시 좌익수가 **이동해서** 잡고 던지는 게 보임. 병살 때 2루 경유 연출. 모든 안무는 같은 경기 재생 시 동일(결정론).

## 3. Track F — 구장 v2 (경기장 퀄리티)

현재: `drawFieldCanvas`가 단일 구장을 절차 드로잉(팔레트 고정, 잠실=사직=고척 동일 그림).

- **F-1. 파크 데이터화**: `KBO_BALLPARKS = { jamsil: { fences: {lf:100,cf:125,rf:100}, fenceHeight, roofed:false, wallColor, seatColors, turfPattern, lightTone } ... }` — 실측 좌우중 펜스 거리로 **비대칭 외야 곡선** 생성(사직 vs 잠실 차이가 보이게), 파울 폴, 워닝트랙 폭. 수치 출처 주석 필수(KBO 구장 공식 제원).
- **F-2. 스킨+조명**: 구장별 팔레트(잔디 톤/담장색/좌석색 — 홈팀 컬러 반영), 잔디 모잉 패턴 2종(체커/스트라이프), **야간 경기 톤**(어두운 관중석+조명탑 광원+필드 하이라이트 — 날짜 해시로 낮/밤 결정, 캘린더 v2 오면 실제 시간대로), 고척 돔은 실내 균일광+천장 라인.
- **F-3. 디테일 레이어**: 전광판에 **실제 이닝/스코어 미니 렌더**(블록 폰트 이미 있음 — blockLetters 재활용), 백스톱 그물 힌트, 더그아웃 안 선수 실루엣 4-5명(대기 타자 배트 들고), 불펜 존 힌트.
- **F-4. 관중 v2**: `attendance`(엔진에 이미 있음) 기반 밀도 — 평일 한산/주말 만원이 보이게. 팀 컬러 응원 구역(홈 3루/원정 1루 KBO 관례 반영해 좌우 배치), 득점/홈런 시 2프레임 웨이브+응원봉 점멸, 치어 단상 1포인트. 정적 텍스처에 굽되 "관중 오버레이 레이어"만 이벤트 시 교체(매 프레임 픽셀루프 금지 — 기존 오프스크린 캐시 원칙 유지).
- 수용 기준: 랩 `?team=`으로 잠실/사직/고척 스킨 차이 스크린샷 확인, 만원/한산 밀도 차이, 야간 경기 톤. `verify:browser` 렌더 체크 유지.

## 4. 우선순위와 의존성

```
1. S-1 파이프라인+애니 플레이어(하위호환)  ← Codex 단독, 소스 없이도 준비 완료 가능
2. D-1 → D-2 수비 루트/포즈               ← Codex 단독, 체감 최대(기존 프레임으로도 효과)
3. F-1 → F-2 구장 데이터화+스킨           ← Codex 단독
4. [유저] 소스 시트 v2 생성 → S-1 완성 → S-2 폼 배리에이션
5. F-4 관중 v2 → D-3/D-4 → F-3 → S-3
```

②등급(소스 재생성)이 도착하기 전에도 1~3으로 "수비수가 움직이고 구장이 달라지는" 체감을 먼저 확보한다.

## 5. 가드레일

1. 경기 결과 로직 무변경. `verify` 3종 + 랩 검사 PASS 유지.
2. 픽셀아트 원칙(기존 스펙 계승): 1px 아웃라인, 플랫 팔레트(액션당 22색 이내, `count_colors` 검사 유지), 안티에일리어싱 금지, 광원 좌상단 고정, 그림자는 엔진 타원.
3. 모든 안무/폼 배정/조명은 deterministic(`hashParts`). 실명·구장 제원은 출처 주석.
4. 필드 텍스처는 오프스크린 1회 캐시(리사이즈 시에만 재생성) — 매 프레임 픽셀 루프 금지(과거 실측으로 멈춤 확인된 안티패턴).
5. 아틀라스 규격 변경 시 `docs/gamecast-sprite-art-spec.md` 동시 갱신 + `reports/motion-samples/` GIF 재출력.
6. `prefersReducedMotion`: 관중 웨이브/점멸 등 반복 모션 오프.
