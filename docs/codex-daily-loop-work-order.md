## 0. 목표 한 줄

지금은 "다음 날"을 누르면 가짜 진행바 5칸이 지나가고 로그 몇 줄이 쌓인 채 끝난다. 목표는 FM처럼: **아침에 편지함이 차 있고, 결재할 게 있고, 오늘 내 경기가 기다리고, 저녁에 그 여파가 편지로 돌아오는 하루.** 유저가 "시간을 넘긴다"가 아니라 "하루를 산다"고 느껴야 한다.

### 목표 체감 시나리오 (이대로 재현되면 성공)

> **4월 14일 화요일 아침.** 클럽하우스에 착지하면 "오늘의 데스크"가 뜬다. 새 편지 5통(안읽음 뱃지 5). ① 비서의 어제 경기 보고(7-3 승), ② 전력분석팀의 오늘부터 시작하는 롯데 3연전 스카우팅 리포트("상대 예상 선발 좌완, 우리 좌타 라인 조심"), ③ 트레이닝 파트 정밀검진 결과 — **[결재] 주전 2루수 햄스트링 10일 이탈, 말소+콜업 / 관찰 / 강행**, ④ 수석코치의 불펜 혹사 경고 — **[결재] 오늘 셋업맨 휴식 지시 여부**, ⑤ KBO 사무국 웨이버 공시 통보. 결재 2건을 처리하니 "오늘 경기" 버튼이 눈에 들어온다. 라인업을 한 번 보고 경기 스킵. 저녁 요약: 2-1 승, 5연승 — 내일 아침엔 "5연승, 가을야구권 진입" 기사와 구단주의 짧은 격려 편지가 와 있다.

## 1. 현재 구조 정밀 진단 (왜 "그냥 지나가는가")

이미 만들어진 것이 꽤 많다. 문제는 **모델(메일함), 리듬(콘텐츠), 멈춤(Continue), 착지(UX)** 네 군데다.

### 1.2 근본 문제 4개

1. **메일함이 없다 — 로그 뷰만 있다.** `state.logs`는 상한 80(engine.js:8)의 문자열/객체 혼합 스택이고, 뉴스함은 이걸 매 렌더마다 파생(`buildNewsInboxItems()` ui.js:1174, 12건 슬라이스). 읽음/안읽음이 **가짜다**: `unreadCount = items.length`(ui.js:1138, 항상 전체 건수), 리스트의 "읽지 않음" 라벨은 index 기반 장식(ui.js:1196). 오래된 편지는 80건 캡에서 증발. → "내 우편함을 관리한다"는 감각이 원천 불가능.
2. **결재가 쌓이지 못한다.** `pendingMailDecision` 단일 슬롯 + `hasOpenMailDecision` 가드 때문에 하루에 결재 2건이 생기면 두 번째는 **생성 자체가 스킵**된다(engine.js:7276, 7352, 7418, 7460, 7489 모두 같은 가드). non-blocking 결재(트레이드 제안, 웨이버)는 빠른 주간에서 답 없이 지나가도 아무 일도 없다 — 세계가 유저 응답을 기다리지 않는다.
3. **멈춤 규칙이 없다.** FM의 심장은 "Continue를 누르면 중요한 일에서 멈춘다"인데, 현재 멈춤은 blocking 의료 결재 하나뿐. 내 팀 경기일에도, 결재 도착에도, 마일스톤에도 빠른 주간은 그냥 지나간다.
4. **하루에 리듬이 없다.** 아침 루틴은 경기 있던 날만 돈다(월요일 휴식일 = 로그 한 줄, engine.js:974). 요일/주간/월간 정기 콘텐츠 없음. 아침 보고 3종이 매일 같은 포맷이라 사흘이면 벽지가 된다. 이벤트(연승, 마일스톤, 순위 레이스, 타 구단 소식) 트리거 없음. 진행 후 착지도 "현재 탭 유지 + status 한 줄"이라 새 편지가 왔다는 신호 자체가 약하다.

### 1.3 타이밍 모델 (오해 방지 — 이건 이미 맞다)

부상은 오늘 경기에서 발생 → 보고 메일은 `morningDate = 경기일+1`로 생성(engine.js:7029) → 유저는 다음 날 아침에 읽고 그날 경기 전에 결재. **시간 순서는 이미 자연스럽다.** 문제는 엔진이 아니라 UI가 "아침"에 착지시키지 않고 경기까지 한 버튼에 묶는 것. 대공사(엔진 페이즈 분리) 없이 UX 오케스트레이션으로 해결 가능하다 — M4 참조.

## 2. 가드레일 (전 WP 공통 — 선행 문서 §2와 동일 + 추가)

1. `npm run verify` / `verify:balance` / `verify:browser` 3종 상시 PASS. 새 시스템은 `tools/verify_app.mjs`에 검증 항목 동봉.
2. 결정론: 모든 확률/선택은 `hashParts`/`deterministicRange` 기반. `Date.now()`/`Math.random()` 금지 (UI 연출용 progressId 제외 관례 유지).
3. **실명 임의 생성 금지.** 스태프/기자에게 사람 이름을 지어 붙이지 말 것 — 현행처럼 역할명("수석코치", "트레이닝 파트", "통역 파트")과 실제 매체명(SPOTV 등)만 사용. 구단주도 "구단주"로.
4. 상태 변경은 엔진 command로만. 메일 읽음 처리도 엔진 함수 경유(저장 대상이므로).
5. `state` shape 추가 시: `normalizeState()`에 마이그레이션(기존 세이브에 mailbox 없음 → 빈 배열 생성), `save.js` roundtrip 검증 통과.
6. `simulateDay`/`simulateDays`/`simulateRegularSeason`의 **기존 시그니처와 "호출하면 하루가 완결된다"는 계약 유지** — verify와 밸런스 스크립트가 이 계약에 의존한다. 페이즈 분리는 내부 하위 단계로.
7. 게임캐스트 파일(`src/gamecastPhaser.js`, `assets/gamecast/*`)은 별도 트랙 — 접근 금지.
8. logs(80캡)는 뉴스 티커/디버그용으로 유지. 메일함은 별도 상태로 분리(M1). 렌더 성능 위해 리스트는 슬라이스 렌더 유지.

## 3. 작업 패키지 (M1 → M8 순서 권장)

### M1. 메일함 모델 v2 — `state.mailbox` (기반, 최우선)

- **문제**: §1.2-1, §1.2-2.
- **스펙**:
  - 새 상태:
    ```js
    state.mailbox = {
      items: [{
        id, date,                    // 결정론적 id (기존 관례: `medical-${date}-${playerId}`)
        from: { role: "트레이닝 파트", icon: "medical" },  // 역할명만, 사람 이름 금지
        category: "club" | "league" | "media" | "decision", // 필터 축
        type: "medical-roster" | "daily-report" | ...,       // 기존 log type 승계
        headline, body,
        read: false,
        decision: null | {           // 결재 첨부 메일
          status: "open" | "resolved" | "expired",
          blocking: false,
          severity: "danger" | "warning" | "notice",
          options: [{ action, label, note }],
          resolvedAction: "", resolvedDate: "", resultMessage: "",
          expiresOn: "",             // 예: 웨이버 = date+7일, 트레이드 제안 = date+3일
          defaultAction: ""          // 만료 시 자동 적용 (보수적 선택: 관찰/패스/거절)
        },
        links: [{ label: "선수 상세", target: "player:{id}" }]  // 관련 화면 바로가기 (M2에서 소비)
      }],
      unread: 0                      // 파생 캐시 (normalize 시 재계산)
    }
    ```
  - 새 엔진 함수: `deliverMail(state, mail)` (push + unread 갱신 + 기존 `addLog` 겸용 호출), `markMailRead(state, mailId)`, `resolveMailDecision(state, mailId, action)` — **시그니처 확장**. 기존 `resolveMailDecision(state, action)` 호출은 "가장 오래된 open 결재"로 위임해 하위호환.
  - **단일 슬롯 폐지**: 결재 생성부 5곳(engine.js:7276/7352/7418/7460/7489)의 `hasOpenMailDecision` 가드 제거 → mailbox 큐잉. 같은 날 결재 상한 3건(초과분은 다음 날 이월)으로 폭주 방지. `pendingMailDecision`은 "blocking이면서 open인 첫 결재"를 가리키는 **파생 selector**로 대체(저장 필드에서 제거, normalize에서 흡수).
  - **만료 처리**: 매일 아침 루틴에서 `expiresOn < today`인 open 결재에 `defaultAction` 자동 적용 + "기한 만료 자동 처리" 결과 메일 발송. → 빠른 주간으로 지나쳐도 세계가 반응한다.
  - 용량: 시즌 중 상한 400건(초과 시 read+비결재부터 제거), 시즌 롤오버 때 `mailbox.items` 비우고 결재 이력은 기존 `state.mailDecisions` 아카이브에 유지.
  - 기존 `addDailyMorningRoutine`의 `addLog({...})` 객체 호출들을 `deliverMail`로 전환(로그 겸용). 문자열 `addLog`(시스템 로그)는 그대로 둔다.
- **수용 기준**: 하루에 의료+트레이드 제안이 동시에 발생해도 둘 다 메일함에 존재. 웨이버 결재를 7일 방치하면 자동 패스 처리 메일 도착. 저장→로드 후 읽음/결재 상태 보존. verify 추가: "메일함 큐잉/만료/roundtrip".

- **문제**: 로그 파생 카드 나열(ui.js:1134-1199), 가짜 읽음, 필터 없음, 본문 열람 개념 없음.

- **수용 기준**: verify:browser에 "받은편지함 2컬럼 렌더/안읽음 뱃지/결재함 필터" 체크 추가. 편지 열면 뱃지 감소. 모바일 390px에서 overflow 0.

### M3. "계속" 버튼 + 멈춤 규칙 (FM의 심장)

- **문제**: §1.2-3. 진행 버튼이 "다음 날/빠른 주간"이라 유저가 매번 며칠 갈지 스스로 정해야 하고, 세계는 중요한 일이 있어도 멈추지 않는다.
- **스펙**:
  - 엔진: `advanceUntilStop(state, { maxDays = 14 })` — 하루씩 진행하며 매일 아침 **멈춤 조건 평가**, 걸리면 그 날 아침 상태로 반환 `{ stopped: true, reason, date, days }`.
  - 멈춤 조건(우선순위순):
    1. blocking 결재 open (현행 유지)
    2. **내 팀 경기일 아침** (기본 ON — FM처럼 경기일은 반드시 유저 손을 거침)
    3. open 결재 메일 존재 (기본 ON)
    4. 중요 메일 도착: 우리 팀 부상 발생, 마일스톤 달성, 순위 이벤트 (기본 ON)
    5. 페이즈 전환일: 정규 종료/포스트시즌/드래프트/FA 개시 (항상 ON)
  - `state.settings.continueStops = { myGameDay: true, openDecision: true, importantMail: true }` — 시스템 탭에 토글 UI(저장 포함).
  - UI: 상단 기본 CTA를 **[계속 ▶]** 하나로. 보조로 "오늘 하루만" / "다음 내 경기까지" / "7일" 드롭다운 유지. 멈추면 status에 사유 표시("트레이드 제안 결재 대기로 4월 17일에 멈췄습니다").
  - `simulateDays`는 내부에서 재사용하되 기존 export 계약 유지(§2-6).
- **수용 기준**: verify 추가 — ① blocking에서 멈춤(기존), ② 경기일 멈춤 ON이면 연속 진행이 내 경기일마다 멈춤, ③ 모든 멈춤 OFF+무결재면 maxDays 완주. 결정론(같은 세이브에서 같은 지점 멈춤).

    - **새 편지 N통** 미리보기 3건(클릭 → 뉴스함 해당 메일) + 결재 대기 M건 버튼(→ 결재함 필터)

    - 액션 버튼: 결재 남았으면 [결재 처리] 강조 → 없으면 [경기 보기]/[빠른 진행] 강조 → 경기 후 [계속 ▶]
  - 하루의 상태 흐름(UI 레벨): `아침(멈춤 지점) → 결재/편지 → 경기(보기/스킵) → 저녁 요약 → [계속]`. 엔진 페이즈 필드는 두지 않는다 — "오늘 내 경기가 진행됐는가"는 lastGames/일정 데이터로 판별 가능.
  - `SIMULATION_STEPS` 고정 연출은 유지하되 단계 텍스트를 실제 발생 이벤트 수로 치환("새 편지 4통 도착" 등, `captureSimulationSnapshot` diff 재사용).
- **수용 기준**: 진행 후 항상 데스크 착지, 새 편지/결재/오늘 경기 3요소가 한 화면에. verify:browser 체크 추가.

## 4. 구현 순서와 이유

```
M1 (모델) ─┬─ M2 (편지함 UI)     ← 여기까지가 "메일함이 진짜가 되는" 최소 단위. 먼저 배포 가치 있음
           └─ M3 (계속+멈춤)      ← FM 감각의 절반
M4 (아침 데스크 착지)             ← 하루의 시작/끝 프레임
M5 (정기 리듬) → M6 (이벤트) → M7 (캠프) → M8 (결재 다양화)   ← 콘텐츠는 뒤에 얼마든지 증분
```

- M1 없이 콘텐츠(M5~M8)부터 넣으면 80캡 로그에서 증발한다. 반드시 모델 먼저.
- M3의 "내 경기일 멈춤"이 들어가야 M4 아침 데스크가 의미를 가진다(경기일마다 데스크를 보게 되므로).
- 각 M은 독립 커밋 + verify 3종 PASS로 마감. 커밋 메시지에 `daily-loop:` 프리픽스 권장.
