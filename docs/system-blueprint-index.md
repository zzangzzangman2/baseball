# KBO GM 시뮬레이터 시스템 블루프린트 인덱스

작성일: 2026-06-30  
작업 범위: 대상 작업 폴더 기준 `docs/system-blueprint-index.md` 신규 작성  
조사 방식: 사용자 제공 목록과 현재 로컬 프로젝트 파일만 기준. 웹 조사 없음.

## 0. 문서 목적

이 문서는 KBO GM 시뮬레이터의 대형 시스템 목록을 내부 기획 인덱스로 정리한다. 각 시스템은 구현 우선순위, 선행 의존성, 필요한 데이터, 현재 프로젝트 상태와의 연결 메모를 함께 둔다.

현재 앱은 정적 HTML/CSS/JS 기반의 Electron 패키징 가능한 skeleton이다. `package.json` 기준 Electron 실행/패키징 구조가 있고, `README.md` 및 검증 보고서 기준 현재 플레이 로스터는 KBO 10개 구단 531명이다. 선수 능력치는 KBO 공식 2026 기록 수집 데이터와 조인되어 있으며, 현재 구조는 공통 16개 + 투수 13개 + 타자 18개, 총 47개 FM식 1~20 능력치와 0~200 OVR/POT를 가진다.

중요한 전제:

- 현재 앱은 MVP 단계다. 2026-03-01 프리시즌 시작, 구단 선택, 하루 단위 진행, 10팀 총 720경기 정규시즌을 deterministic하게 진행할 수 있다. P0 완료 후 기준은 한국시리즈 이후 autonomous offseason v1로 진입해 FA/외국인 권리, 드래프트/2차 드래프트 반영, 다음 시즌 rollover를 처리하는 것이다. 실제 일정/엔트리/장기 저장 계층은 이후 고도화 대상이다.
- 트레이드와 FA/외국인 시장은 v1/v2 command로 실제 상태를 commit한다. autonomous offseason v1도 같은 command/result ledger 원칙을 따른다. 스카우트/프런트오피스의 나머지 화면 데이터는 아직 selector와 제안 생성에 가깝다.
- 실제 로스터 1,052명, 2025 기록, 계약, 병역, FA, 외국인, 연봉 등은 목표 데이터셋이다. 현재 플레이 로스터 531명과 혼동하지 않는다. 외국인 시장은 공식 실명 데이터가 들어오기 전까지 `FGN-YYYY-###` 코드형 권리로만 유지한다.
- 임시 가정: 1,052명 목표 로스터는 현역 1군/퓨처스뿐 아니라 육성, 군보류, 신인, 보류권, 외국인 후보 일부까지 포함하는 장기 원장으로 본다.

## 1. 우선순위 기준

| 우선순위 | 의미 | 대표 작업 |
| --- | --- | --- |
| P0 | 모든 대형 시스템의 공통 기반. 먼저 없으면 뒤 기능이 서로 상태를 오염시킬 위험이 큼. | 단일 GameState, transaction, eventLog, 저장 포맷, 선수 원장, autonomous offseason v1, season rollover, draft roster assignment |
| P1 | 현재 MVP에 바로 체감되고 데이터 기반이 이미 일부 있는 핵심 확장. | 로스터 DB 확장, 능력치 registry, 시즌 캘린더, 라인업/로테이션 |
| P2 | 시즌 진행의 신뢰도를 올리는 중형 시스템. | 타석 단위 엔진, 밸런스 튜닝, 부상/피로, 포스트시즌/시상 |
| P3 | 스토브리그와 프런트 운영. 선행 데이터가 꽤 필요함. | 트레이드, FA/외국인, 드래프트, 2차 드래프트, 계약 |
| P4 | 장기 몰입과 서사. autonomous offseason v1이 안정된 뒤 full multi-year 운영으로 확장. | 재정, 뉴스/여론, 회의/정치, 노화/은퇴/시설/팬베이스 |

## 2. 공통 기반 인덱스

대형 시스템은 기능별로 따로 붙이면 금방 결합이 폭발한다. 특히 스카우트, 뉴스, 여론, 재정, 계약, 트레이드는 같은 선수/팀 상태를 동시에 읽고 쓰기 쉽다. 먼저 아래 기반을 만들고, 개별 시스템은 이 기반에 연결하는 방식이 안전하다.

| 기반 | 우선순위 | 역할 | 현재 상태 연결 메모 |
| --- | --- | --- | --- |
| 단일 `GameState` | P0 | 리그/팀/선수/일정/계약/이벤트의 유일한 진실 | 현재 `createInitialState()`가 팀 배열과 로스터를 만든다. 장기적으로 `playersById`, `teamsById`, `calendar`, `contractsByPlayerId`로 정규화 필요 |
| Transaction/Command | P0 | 모든 변경을 검증 후 commit | 트레이드 v2, FA 계약, 외국인 보류권 계약은 command로 검증 후 commit한다. P0 완료 후 기준에서는 오프시즌 단계 진행, 드래프트 코드형 권리/입단 예정 반영, 2차 드래프트 소속 변경, season rollover도 같은 원칙으로 처리한다. 보상선수/다자산/재정 지급은 후속 확장 |
| Event Log | P0 | 경기, 계약, 부상, 뉴스, 회의 결과의 원천 기록 | `state.eventLog`가 경기, 트레이드, FA/외국인 계약 이벤트를 남긴다. autonomous offseason v1은 단계 완료와 season rollover 이벤트를 남기는 기준이며, 뉴스와 완전 리플레이 가능한 event schema는 더 확장해야 함 |
| 저장 포맷 | P0 | JSON 저장, 이후 SQLite 전환 기반 | 현재 `src/save.js`는 JSON export/import 순수 함수. P0 완료 후 season rollover 저장 roundtrip을 검증 기준에 포함한다. 자동 localStorage는 피해야 함 |
| 데이터 출처/신뢰도 | P0 | 공식 기록, 후보, fallback 구분 | 현재 ratingSource와 sourceUrls가 일부 존재. 1,052명 확장 시 후보/확정 상태를 분리해야 함 |
| Attribute Registry | P1 | 47개에서 60개 확장 시 버전/스케일 관리 | 현재 `ratings.js`에 47개 능력치 배열과 weight가 있다. 새 능력치는 registry/migration 필요 |

## 3. 시스템별 기획 인덱스

### 3.1 실제 로스터 DB

요구 범위:

- 10개 구단 1,052명
- 2025 기록, 계약, 병역, FA 상태, 외국인 여부, 연봉
- 수비/주루/투수 세부 스탯

구현 우선순위: P1  
선행 의존성: 선수 원장, 출처 신뢰도, 저장 schemaVersion, 팀/선수 id 정책  
현재 상태:

- 현재 플레이 로스터는 10개 구단 531명이다.
- KBO 공식 등록/퓨처스 중심 로스터가 앱에 들어가 있으며, 후보 확장 데이터는 별도 후보 파일로 관리된다.
- 530명은 KBO 공식 2026 정규/퓨처스 상세기록 기반 ratingSource를 가진다.

필요 데이터:

- 2025 정규/퓨처스 기록 원본
- 계약 기간, 연봉, 계약금, 옵션, 보상/FA 등급
- 병역/군보류/육성/방출/FA/외국인 status
- 수비 이닝, 포지션별 출장, 실책 외 수비 지표
- 주루, 도루 성공률, 견제/도루 저지, 투수 구종/구속/타구 성향

구현 메모:

- 목표 1,052명은 바로 플레이 로스터에 섞지 말고 `playersById` 원장에 넣은 뒤 `rosterAssignments`로 active/futures/candidate/military/fa를 구분한다.
- 2026 능력치와 2025 기록을 동시에 다룰 수 있도록 `statsBySeason[playerId][season]` 구조가 필요하다.
- 임시 가정: 현재 531명은 MVP 플레이 가능 로스터로 유지하고, 1,052명 원장은 장기 플레이/드래프트/2차 드래프트/군보류/FA 시스템에서 단계적으로 사용한다.

### 3.2 선수 능력치

요구 범위:

- FM식 1~20 능력치
- 0~200 OVR/POT
- 타자/투수/공통 능력치 47개에서 이후 60개까지 확장

구현 우선순위: P1  
선행 의존성: Attribute Registry, 능력치 산정 버전, 데이터 출처/신뢰도  
현재 상태:

- 현재 `ratings.js`는 공통 16개, 투수 13개, 타자 18개를 제공한다.
- 공식 KBO 2026 기록 기반 산정과 fallback이 섞여 있고, 표본 신뢰도를 함께 둔다.
- OVR/POT는 0~200 범위로 검증된다.

필요 데이터:

- 타자 좌우 split, 타구질, 수비 범위, 송구, 포수 수비
- 투수 구속, 구종, 움직임, 땅볼 성향, 피홈런 억제, 좌우 상대
- 멘탈/리더십/워크에식은 공식 기록만으로 부족하므로 스카우팅/코칭/평판 데이터 필요

구현 메모:

- 47개 능력치 배열을 단순 확장하면 UI, OVR weight, POT, 저장 호환성이 같이 깨질 수 있다.
- `attributeDefinitions`에 key, label, role, scale, category, sourcePolicy, version을 둔다.
- 60개 확장 시 기존 저장본 migration과 능력치 표시 UI를 같이 처리한다.

### 3.3 시즌 시뮬레이션

요구 범위:

- 10팀 144경기
- 타석 단위 결과
- 부상, 컨디션, 피로, 사기, 팔 상태
- 경기장, 날씨, 관중 반영

구현 우선순위: P1~P2  
선행 의존성: 캘린더, 라인업/로테이션, 선수 상태, eventLog, 누적 기록 ledger  
현재 상태:

- 현재 엔진은 2026-03-01 프리시즌부터 개막일까지 경기 없이 하루씩 진행하고, 정규시즌 총 720경기를 deterministic하게 끝까지 진행한다.
- 실제 일정표가 아니라 생성 매치업이며, 타석 v1, 이닝별 line score, 선발/불펜 운용, 선수 누적 기록, 부상/피로/관중을 간단히 반영한다.
- UI에서는 빠른 주간/시즌 전체 진행 버튼을 제거하고 `다음 날` 진행만 노출한다.

필요 데이터:

- 실제 또는 생성형 시즌 캘린더
- KBO 홈구장 park factor, 날씨 factor, 관중/흥행 factor
- 선수별 타격/투구/수비/주루 세부 능력치
- 투수 팔 상태, 등판 간격, 불펜 소모, 포수/야수 피로

구현 메모:

- 바로 타석 엔진으로 뛰기 전에 `CalendarGame`, `GameRoster`, `LineupSnapshot`, `PitchingPlanSnapshot`을 먼저 만든다.
- 타석 결과는 `plateAppearanceEvents`로 남기고, 좌우 split/타구유형/수비/실책/병살/구장 컨텍스트를 함께 싣는다.
- 초기 P2 목표는 실제 야구 완전 재현이 아니라, 리그 평균에 맞는 결과와 로스터 운영 피드백을 만드는 것이다.

### 3.4 밸런스 튜닝

요구 범위:

- 2024 KBO 평균에 맞춰 타율, 출루율, 장타율, ERA, 득점, 홈런, 도루, 부상 수 검증

구현 우선순위: P2  
선행 의존성: 시즌 시뮬레이션, 기록 ledger, 반복 검증 스크립트  
현재 상태:

- 현재 검증은 앱 구조, 시즌 완주 smoke test, 타석/박스스코어 누적, UI overflow/clipping 중심이다.
- 한 시즌 전체 표본 기준 타율/출루율/장타율/ERA를 KBO 평균권으로 맞추도록 1차 튜닝했고 `tools/verify_balance.mjs`와 `reports/balance.md`로 검증한다. 고정 seed 다회 리그 평균/이상치 리포트는 후속이다.

필요 데이터:

- 2024 KBO 리그 평균 상수
- 팀/구장별 보정값
- 부상 발생 기준치, 평균 결장일, 포지션별 위험도

구현 메모:

- `tools/verify_balance.mjs` 같은 고정 seed 다회 시뮬레이션 검증이 필요하다.
- 지표는 목표값, 허용 오차, 현재값, 편차를 표로 출력한다.
- 타석 엔진 전에는 경기 단위 결과라도 득점/ERA/홈런/도루/부상 발생량을 먼저 맞춘다.

### 3.5 포스트시즌/시상식

요구 범위:

- 와일드카드, 준PO, PO, 한국시리즈
- MVP, 신인왕, 골든글러브 자동 생성

구현 우선순위: P2  
선행 의존성: 정규시즌 종료 판정, 순위 타이브레이커, 기록 ledger, eventLog  
현재 상태:

- 현재 `initializePostseason(state)`가 정규시즌 상위 5팀 seed와 4개 시리즈 bracket을 만든다.
- 현재 `simulatePostseason(state)`는 와일드카드, 준PO, PO, 한국시리즈를 끝까지 진행하고 `phase=offseason`으로 넘긴다.
- 시상식 v1은 정규시즌 MVP, 신인왕, 골든글러브 10명, 한국시리즈 MVP를 자동 산출한다.

필요 데이터:

- KBO 포스트시즌 진출/어드밴티지/시리즈 규칙
- 선수 누적 기록, 포지션별 출장, 신인 자격, 투표/가중치 기준

구현 메모:

- v1은 KBO 운영제도 기준으로 와일드카드 4위 1승 어드밴티지, 준PO/PO 5전3선승, 한국시리즈 7전4선승을 사용한다.
- 포스트시즌 경기는 정규시즌 팀 승패를 변경하지 않고 `state.postseason.series[].games`와 최근 경기 목록에만 남긴다.
- 시상식은 실제 투표가 아니라 누적 기록 기반 산출식이다. 공식 신인 자격/포지션별 수비 이닝/투표 규칙은 후속 고도화 대상이다.
- 장기적으로는 postseason game event와 regular-season `seasonStats`를 분리하고, 포스트시즌 전용 기록/수상 근거를 따로 저장해야 한다.

### 3.6 드래프트

요구 범위:

- 신인 드래프트 11라운드
- 150명 유망주 풀
- 학력별 성장 차이
- 20-80 스카우팅
- AI 지명 전략

구현 우선순위: P0 autonomous offseason v1, P3 실명/계약 고도화  
선행 의존성: 선수 원장, 스카우트 리포트, 지명권 ledger, 장기 시즌 순환  
현재 상태:

- `initializeDraft(state)`와 `simulateDraft(state)` v1이 있다.
- 정규시즌 역순 기반 전면 드래프트 순서, 11라운드 110픽, 150명 후보 풀을 생성한다.
- 후보는 `DRF-YYYY-###` 익명 코드와 `anonymous-projection-v1` 출처를 사용한다. 실명 검증 전 신인 이름을 만들지 않는다.
- 팀별 roster depth를 읽어 즉전/업사이드/균형 전략, 선호 포지션, 리스크 선호를 만든다.
- P0 완료 후 autonomous offseason v1 기준에서는 지명 결과를 `DRF-YYYY-###` 코드형 권리/입단 예정 레코드로 다음 시즌 `rosterAssignments`에 반영한다.
- 이 반영은 공식 실명 선수를 새로 등록했다는 뜻이 아니다. 실명, 생년월일, 학교, 계약금, 실제 계약 상태는 공식 데이터 수집 전까지 비어 있거나 코드형 상태로 남긴다.
- 프런트오피스 스카우트 업무에서 "국내 신인/퓨처스 보강 방향"은 아직 별도 리포트/오차범위와 연결되지 않았다.

필요 데이터:

- 고교/대학/해외/얼리 후보 풀
- 포지션, 생년월일, 투타, 학교, 신체조건
- 현재치/잠재치 20-80, 리스크, 성장 유형
- 팀별 needs, 성향, 순위, 보상/트레이드 지명권

구현 메모:

- 임시 가정: 150명 풀에서 11라운드 지명을 진행하고 미지명 40명은 독립/대학/재수/육성 후보로 남긴다.
- 실제 선수명이 검증되지 않은 경우 이름 생성은 금지하고 익명 후보 코드로 명시해야 한다.
- 20-80 스카우팅은 실제 능력치를 공개하는 UI가 아니라 불확실성을 줄이는 리포트로 사용한다.
- 후속 작업은 공식 신인 명단 수집, 실명/생년월일/학교/계약금 등록, 실제 계약 command, 지명권 트레이드 ledger, 스카우트 리포트 불확실성 고도화다.

### 3.7 2차 드래프트

요구 범위:

- 35인 보호명단
- 자동 보호
- AI 지명
- 지명 거부

구현 우선순위: P0 autonomous offseason v1, P3 규칙 고도화  
선행 의존성: 전체 선수 원장, rosterAssignments, 계약/보류권, AI 가치 평가  
현재 상태:

- `initializeSecondaryDraft(state)`와 `simulateSecondaryDraft(state)` v1이 있다.
- 현재 531명 플레이 로스터 기준으로 구단별 35인 보호명단, 비보호 풀, 하드 제외 목록을 만든다.
- 지명 순서는 정규시즌 역순이며 1~3라운드는 10개 구단, 4~5라운드는 하위 3팀만 추가 지명한다.
- 원소속팀 피지명은 최대 4명으로 제한하고 라운드별 양도금을 결과에 저장한다.
- P0 완료 후 autonomous offseason v1 기준에서는 `state.secondaryDraft.picks` 결과를 다음 시즌 소속 `rosterAssignments`에 반영한다. 다만 지명 거부, 의무등록, 보상금의 실제 재정 지급은 후속 command로 남긴다.
- 외국인/FA 시장 선수는 hard-excluded 처리한다. 입단연차 자동 제외는 현재 추정 데이터로 엄격 적용하면 MVP 로스터의 비보호 풀이 과도하게 줄어들어, 1,052명 원장 확장 후 공식 취득연차로 적용한다.

필요 데이터:

- 2차 드래프트 대상/제외 규칙
- 연차, 군보류, 육성, FA, 외국인, 신인 보호 여부
- 팀별 35인 보호명단 가치 평가

구현 메모:

- 자동 보호는 OVR/POT만 보면 안 된다. 포지션 scarcity, 계약, 나이, 군 문제, 팀 timeline을 반영한다.
- 지명 거부는 선수 status와 계약/보상 처리까지 연결되어야 한다.
- v1의 보호 점수는 OVR/POT, 나이, 포지션 희소성, 연봉 부담을 반영한다.
- 후속 작업은 보상금 ledger의 실제 재정 반영, 의무등록/지명 거부, 군보류/육성/입단연차 공식 데이터 연결이다.

### 3.8 트레이드

요구 범위:

- 선수, 현금, 지명권, 조건부 자산, 후일결정선수
- 선수 가치, 지명권 가치, AI 수락 판단

구현 우선순위: P3  
선행 의존성: transaction, 계약, 지명권 ledger, 선수 원장, AI 팀 전략  
현재 상태:

- `src/frontOffice.js`는 트레이드 시장, 타깃, 패키지 제안, 수락 가능성을 deterministic하게 만든다.
- `commitTradeProposal(state, proposal)` v2는 수락 가능한 선수 1명+보조 자산 패키지를 검증 후 commit한다.
- commit 시 양 팀 roster에서 선수를 교환하고, `player.teamId`, `contract.teamId`, FA 권리팀, 군보류팀, 외국인 권리팀을 갱신한다.
- 현금, 지명권, 조건부 자산, 후일결정선수는 `tradeAssets.cashLedger`, `draftPickLedger`, `conditionalAssets`, `ptbnlSlots`에 남긴다.
- `state.trades.completed[]`와 `eventLog`의 `trade.completed` 이벤트에 실행 결과를 남기며 저장 roundtrip 검증을 통과한다.
- 선수 2명 이상이 오가는 다자산 패키지와 실제 재정 예산 반영은 아직 없다.

필요 데이터:

- 선수 계약/연봉/서비스타임/부상/폼
- 지명권 소유권과 가치 모델
- 현금/연봉 부담, 조건부 트리거, 후일결정선수 deadline

구현 메모:

- 현재 v2는 `commitTradeProposal` 단일 함수지만, 후속에서는 `PROPOSE_TRADE`, `ACCEPT_TRADE`, `COMMIT_TRADE` command로 나눈다.
- PTBNL은 `conditionalAsset`으로 만들고, 조건 충족 시 후속 transaction을 예약한다.
- AI 수락 판단은 값 균형뿐 아니라 팀 전략, 예산, 팬 여론, 구단 정치까지 확장 가능하게 둔다.

### 3.9 FA/외국인 시장

요구 범위:

- FA 등급, 보상
- 계약금/연봉/인센티브/옵션/트레이드 거부권
- 에이전트
- 외국인 5티어 풀

구현 우선순위: P0 autonomous offseason v1, P3 시장 고도화  
선행 의존성: 계약 시스템, 연봉/재정, 선수 status, 스토브리그 phase  
현재 상태:

- 선수별 `contract`, `faStatus`, `foreignPlayer`, `compensationGrade` v0 schema가 있다.
- `initializeFreeAgency(state)`, `commitFreeAgentSigning(state, offer?)`, `commitForeignPlayerSigning(state, offer?)`가 구현되어 국내 FA roster 이동/계약 갱신/보상 ledger와 외국인 코드형 보류권 ledger를 commit한다.
- UI에서는 FA/외국인 시장을 표시하되, 정규시즌 도중 즉시 스토브리그가 열리지 않도록 시즌 완료 이후로 action을 guard한다.
- P0 완료 후 autonomous offseason v1에서는 FA 계약 결과와 외국인 코드형 권리를 season rollover에 포함한다. 외국인은 공식 실명/국적/계약 데이터가 들어오기 전까지 실제 roster player로 등록하지 않는다.

필요 데이터:

- FA 자격, 등급, 보상 규칙, 보호선수 명단
- 계약 구조: 보장액, 옵션, 인센티브, 기간, NTC
- 외국인 선수 pool, 국적, 포지션, 최근 리그/성적, 몸값 tier
- 에이전트 성향, 협상 난이도, 선호 팀/조건

구현 메모:

- `state.freeAgency`는 `{ status, year, ruleSource, sourceUrls, faCandidates, offers, foreignMarket, foreignOffers, signings, compensationLedger, foreignSignings }`를 둔다.
- 국내 FA 후보는 실제 로스터 DB의 국내 선수 중 `faStatus`와 `compensationGrade` 기준을 만족하는 선수만 사용한다.
- FA 계약 command는 선수 `teamId`, `contract`, `faStatus`를 갱신하고 보상 결과를 `compensationLedger`에 남긴다.
- 보상 룰 v1은 A=20인 보호+전년도 연봉 200% 또는 300% 현금, B=25인 보호+100% 또는 200% 현금, C=150% 현금/보상선수 없음이다.
- 외국인 시장은 검증된 실명이 아니므로 `FGN-YYYY-###` 코드형 30명/5티어 후보로만 만든다.
- 외국인 계약 command는 roster에 가짜 이름을 넣지 않고, 보류권/활성화 대기 ledger만 생성한다.
- 계약은 재정 시스템보다 먼저 최소 shape를 둔다. 실제 지급/예산 반영은 후속 `financeLedger`에서 처리한다.

선행 의존성: eventLog, 팀/구단주 성향, 장기 규정 변경, 뉴스/여론  
현재 상태:

- 미구현이다.
- 현재 게임에는 규정 변경과 정치적 자본 개념이 없다.

필요 데이터:

- 구단별 이해관계, 시장 규모, 재정 상태, 경쟁력, 구단주/단장 성향
- 안건 템플릿: 외국인 확대, 엔트리, 드래프트, CBT, 일정, 포스트시즌
- 투표 규칙, 통과 조건, 회의록 생성 규칙

구현 메모:

- 이 시스템은 장기 플레이의 규정 변경 장치로 쓰는 편이 좋다.
- 초기에는 실제 단체/인물 이름을 생성하지 말고 구단 단위 익명 회의록으로 시작한다.
- 회의 결과는 `leaguePoliciesByDate` 또는 다음 시즌 rule set으로 반영한다.

### 3.15 뉴스/여론 시스템

요구 범위:

- 기사, 속보, 루머, 기자 성향, 댓글 유형
- 여론 7차원
- 이벤트 체인
- 구현 일부 placeholder 위험

구현 우선순위: P4  
선행 의존성: eventLog, 팀/선수 평판, 경기/계약/부상/정치 event, 텍스트 템플릿 검증  
현재 상태:

- 현재 앱에는 로그, 프런트오피스 inbox, marketNotes가 있지만 본격 뉴스/여론 시스템은 없다.
- placeholder/name-generator 잔재는 검증에서 차단하고 있다.

필요 데이터:

- 뉴스 템플릿, 기자 성향, 매체 타입
- 여론 7차원 정의. 예: 성적 만족, 프런트 신뢰, 선수 호감, 감독 신뢰, 재정 인식, 미래 기대, 논란 민감도
- 루머 신뢰도, 댓글 유형, 이벤트 체인 조건

구현 메모:

- placeholder 위험이 큰 시스템이다. 검증되지 않은 실제 선수/기자/매체 이름을 만들지 않는다.
- 뉴스는 eventLog에서 파생한다. 독립적으로 사실을 만들어내면 저장/리플레이와 충돌한다.
- 루머는 `rumor` 타입으로 명시하고, 실제 확정 사건과 UI에서 구분한다.
- 댓글은 요약된 여론 샘플로 표현하고 혐오/실명 공격성 텍스트를 피한다.

### 3.16 장기 플레이 계획

요구 범위:

- 노화, 은퇴, 매년 드래프트 클래스
- FA 갱신, 외국인 풀 갱신
- 규정 변경, 시설 노후화, 팬베이스 변동

구현 우선순위: P0 autonomous offseason v1, P4 full multi-year 고도화  
선행 의존성: 시즌 종료 pipeline, 선수 원장, 계약/FA, 드래프트, 재정, eventLog, 저장 migration  
현재 상태:

- P0 완료 후 기준은 full 장기 운영이 아니라 autonomous offseason v1이다.
- 한국시리즈 종료 후 오프시즌 단계가 순차 실행되고, FA/외국인 권리 처리, 신인 드래프트 코드형 권리, 2차 드래프트 소속 변경, 다음 시즌 프리시즌/정규시즌 준비까지 한 번의 season rollover로 연결한다.
- 은퇴, 노화/성장, 완전한 FA 갱신, 공식 신인/외국인 실명 수집, 시설/팬베이스 변동은 아직 v1 범위 밖이다.

필요 데이터:

- 연령/포지션별 성장/노화 곡선
- 은퇴 확률, 부상 누적, 성적 하락, 계약 만료
- 매년 드래프트 클래스 생성/수집 정책
- 외국인 풀 갱신, 시설 수명, 팬베이스 변동 모델

구현 메모:

- 장기 플레이는 모든 시스템의 최종 통합 지점이다.
- `advanceOffseasonStage()`를 작은 단계로 쪼갠다. 예: awards, FA filing/signings, foreign rights, amateur draft rights, secondary draft assignments, contract rollover, spring setup.
- P0 v1은 한 시즌을 다음 시즌으로 넘기는 autonomous league loop를 만드는 데 집중한다. 다년 재정, 뉴스/정치, 선수 노화/은퇴, 실명 외국인 등록은 별도 고도화로 둔다.
- 저장본 migration과 event replay가 불안정하면 장기 플레이에서 버그가 누적된다.

## 4. 의존성 요약

가장 위험한 결합:

- 스카우트 + 선수 능력치: 스카우트 리포트가 실제 능력치를 직접 바꾸면 데이터 오염이 생긴다.
- 뉴스/여론 + eventLog: 뉴스가 독립적으로 사실을 만들면 저장/리플레이와 충돌한다.
- 트레이드/FA/드래프트 + 로스터 DB: 선수 이동이 transaction 없이 여러 배열을 직접 수정하면 중복/실종 선수가 생긴다.
- 재정 + 계약 + 관중: 수입/지출 ledger 없이 payroll만 수정하면 장기 저장에서 원인 추적이 안 된다.
- 장기 플레이 + migration: 능력치/계약/상태 schema가 바뀔 때 저장본 호환성이 깨질 수 있다.

P0 완료 후 권장 구현 순서:

1. autonomous offseason v1의 단계 순서, idempotency, 저장 roundtrip, season rollover 검증을 고정한다.
2. 현재 531명 로스터를 보존하면서 1,052명 목표 원장 구조와 `rosterAssignments`의 공식 데이터 보강 경로를 설계한다.
3. 드래프트 코드형 권리와 외국인 코드형 권리를 실제 실명 선수 등록과 분리하는 activation contract를 만든다.
4. 47개 능력치를 registry화하고 60개 확장 migration을 준비한다.
5. 시즌 캘린더, 라인업/로테이션, 투수 사용량을 개선하고 이후 트레이드/FA/드래프트 고도화를 붙인다.
