## 1. 질문 ①: "60fps까지 뽑아야 하나?" — 답: 프레임레이트는 이미 60 타깃이다. 병목은 fps가 아니라 **모션의 질**이다.

구조적 사실:
- Phaser 설정이 이미 `fps: { target: 60 }`이고(gamecastPhaser.js:87), 애니메이션은 `elapsedMs` 기반 연속 보간이라 프레임레이트가 높아지면 자동으로 부드러워지는 구조다. easeOutCubic/easeInOutCubic/easeOutBack 이징도 이미 쓰고 있다(ui.js:6498, 6996 등).
- 즉 "60fps로 올리는 작업"은 존재하지 않는다. 대신 **60fps를 깎아먹는 것**과 **60fps여도 어색해 보이게 만드는 것**을 제거해야 한다.

어색함의 실제 원인 5개 (우선순위순):

1. **무음 중계.** 사운드 코드가 0줄이다(`Audio` 참조 전무). 타격음·포구음·관중 소리가 없으면 같은 모션도 훨씬 어색해 보인다 — 최소 비용에 최대 체감.
2. **매 프레임 스프라이트 전체 파괴/재생성.** `renderScene`이 프레임마다 `spriteLayer.removeAll(true)` 후 `scene.add.image(...)`를 ~15개씩 새로 만든다(gamecastPhaser.js:306-319). 60fps에서 초당 ~900개 객체 생성 → GC 히치로 간헐적 프레임 드랍(스터터). 스프라이트 풀로 교체.
3. **타석 간 하드컷.** PA 슬롯(2600ms)+갭(340ms) 뒤 다음 PA로 화면이 뚝 끊긴다. 타자 교대 브리지·수비 복귀 모션이 없다.
4. **애니메이션 프레임 부족.** 달리기 2프레임, 스윙 3단계(stance→swing→follow) 등 아틀라스 프레임이 최소 단위(gamecastPhaser.js:4-28). 초당 프레임이 아니라 **포즈 프레임**이 모자란 것.
5. **모든 타석이 같은 템포.** 만루 동점이든 3구 범타든 2600ms 고정 — 리듬이 없어 "돌아가는 배경"처럼 느껴진다.

→ 결론: 아래 Q-1~Q-6이 "더 퀄리티 높게, 자연스럽게"의 실체다. 측정 가능하도록 Q-0(FPS 카운터)부터.

## 2. 질문 ②: "경기 진행 방식 이대로 좋을까?" — 답: **연출은 유지, 엔진 구조는 2단계로 교체 권고.**

### 현재 구조의 진실 (중요)

`simulateGame`(engine.js:3504)은 **원정팀 공격 9이닝 전체를 한 번에, 홈팀 공격 9이닝 전체를 또 한 번에, 서로 독립적으로** 시뮬레이션한다(`simulateOffense` 2회 호출, engine.js:3521-3543). 게임캐스트는 그 결과 이벤트를 이닝/사이드로 **정렬해서 인터리브처럼 보여줄 뿐**이다(`sortGamecastEvents` ui.js:4722).

이 구조로는 원천적으로 불가능한 것:
1. **끝내기(walk-off)** — 홈팀은 9회말에 이기고 있어도 끝까지 다 친다(홈 공격이 원정 결과를 모른 채 시뮬되므로). 야구 드라마의 정점이 통째로 없음 + 홈팀 타석 수가 현실 대비 과다.
2. **점수 상황 인지 불펜 운용** — 세이브 상황에 마무리 투입 같은 판단이 사후 근사(pitcherDecisions 라벨링)일 뿐, 시뮬 자체는 상대 점수를 모른다.
3. **경기 중 감독 개입** — 6회 위기에서 투수 교체 지시 같은 FM식 감독 모드가 구조적으로 불가.
4. **진짜 연장전** — 포스트시즌 10회 강제 보정(`forcePostseasonDecision`)으로 때우는 중. 선행 문서 WP-07(연장 12회/무승부)도 이 구조 위에서는 어색하게 붙는다.

유지의 장점도 분명하다: 단순·빠름·결정론 검증 파이프라인 안정. 그래서 **한 번에 갈아엎지 말고 단계 이행**:

### Stage A — 지금: 방식 유지 + 연출 품질 (이번 지시서의 Q 시리즈)
연출 계층은 이벤트 리플레이 설계 그대로 두고 Q-0~Q-8을 진행한다. 엔진 무변경.

### Stage B — 엔진: 인터리브 이닝 루프 (WP-LIVE-1)
- `simulateGame`을 하프이닝 단위 인터리브 루프로 재작성: `for inning { away half; home half(9회말 리드 시 스킵/끝내기 중단); }` + 연장 12회(정규)/15회(포스트시즌) 자연 지원 — **선행 문서 WP-07과 반드시 동시 진행**(같은 코드를 두 번 손대지 않기).
- 타석 모델(`resolvePlateAppearance`)과 투수 교체 로직은 그대로 재사용. 시드는 하프이닝 단위로 파생(`hashParts(gameSeed, inning, side)`)해 결정론 유지.
- 불펜 판단에 실제 점수차/이닝(리버리지) 입력 추가 — 마무리는 진짜 세이브 상황에 나온다.
- **리스크**: 타석 분포가 달라져(9회말 스킵, 끝내기 중단) 리그 총 타석·득점이 ~2-4% 감소한다. `verify:balance` 기준(AVG .278/ERA 4.64/HR 1.93)을 재통과할 때까지 확률 계수 재튜닝이 한 세트다. 이 재튜닝 때문에 "한 번에, 별도 커밋 시리즈로" 진행할 것.
- 수용 기준: 한 시즌에 끝내기 승리 20~60회(현실 KBO 연 40회 내외), 홈/원정 팀당 타석 비대칭이 현실 방향, balance PASS, 기존 verify 전부 PASS.

### Stage C — 라이브 세션 = 감독 모드 (WP-LIVE-2, Stage B 선행 필수)
- `state.liveGame` 세션: 하프이닝을 **하나씩 progressive 시뮬**하고 게임캐스트가 방금 시뮬된 하프이닝만 리플레이. 이닝 교대 때 결정 창 — 투수 교체, 대타/대주자 예약, (후속) 번트/도루 지시. 이미 있는 `buildGameInterventionForTeam`/`applyPitchingIntervention` 훅(engine.js:3510-3513)을 세션 결정으로 확장.
- 결정이 다음 하프이닝 시드 입력보다 먼저 상태에 반영되므로 결정론 유지. "관전 안 함" 경로는 결정 없이 전체 자동(현행과 동일 결과 경로).
- 이게 완성되면 "실시간 경기를 지휘한다"는 FM 감각이 진짜가 된다. 유저 개입 없이 보기만 할 수도 있음(현행 UX 하위호환).

**권고**: Stage A는 즉시, Stage B는 다음 엔진 사이클에서 WP-07과 묶어 착수, Stage C는 B 안정화 후. A와 B는 파일 겹침이 적어 병행 가능.

## 4. 가드레일

1. 경기 **결과** 로직 무변경(Stage A 전체). `verify`/`verify:balance` 무영향. Stage B만 예외이며 별도 커밋 시리즈+재튜닝 세트.
2. 연출 랜덤(카메라 미세 흔들림, 문장 선택, 사운드 피치 변주)은 `hashParts(eventId, ...)` 기반 — 같은 경기 재생 시 같은 연출.
3. `prefersReducedMotion` 존중: 카메라 워크/셰이크/플래시 오프, 사운드는 별개 토글.
4. 사운드는 유저 제스처(재생 시작 버튼) 이후에만 AudioContext resume — autoplay 정책 위반 콘솔 경고 금지.
5. 랩(`gamecast-lab.html`) 기준 검증 유지. verify:browser의 랩 체크가 계속 PASS해야 한다.
6. 스프라이트 아틀라스 규격 변경 시 `docs/gamecast-sprite-art-spec.md` 갱신.
