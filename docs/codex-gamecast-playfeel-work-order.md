# Codex 작업지시서: 관전 체감(Play-Feel) — "야구가 보이게 하라"

작성일: 2026-07-07
판정 방법: 실제 플레이어처럼 관전 — 헤드리스로 랩(`?holds=0`) x1 배속을 **420ms 간격 16프레임 연속 캡처**해 타석 2개(구자욱 홈런, 최형우 내야 아웃)의 공 흐름/선수 흐름을 프레임 단위로 추적. 캡처 스크립트 패턴은 `tools/capture_gamecast_shots.mjs` 참고(버스트 모드 추가 지시 포함, §4).

## 0. 플레이어 시점 총평

구장·스프라이트·HUD·피드는 좋아졌는데, 정작 **야구의 동사(動詞)가 화면에 없다.** 16프레임(약 6.7초, 타석 2개 분량) 동안:
- **공이 단 한 프레임에도 안 보였다.** 투구 궤적도, 홈런 타구 비행도 관찰 불가.
- **홈런이 "글자"로만 존재한다.** "홈런!" 배지가 뜨고 점수가 0→1로 바뀌는데, 타구도 주자 베이스 러닝도 없음.
- **내야 아웃인데 수비수가 아무도 안 움직였다.** 9명 전원 제자리, 결과 배지("IF 아웃")만 매치업 칩에 표시.
- **마운드에 투수가 없다.** 투구 모션 자체가 없으니 타석이 "정지화면 → 결과 글자"로 소비된다.
즉 지금 관전은 야구 중계가 아니라 **필드 배경 위에 뜨는 자막 방송**에 가깝다. 이걸 고치는 게 이번 지시서다.

## 2. 수리 목록 (PF 시리즈 — 위에서부터)

### PF-1. 타석 = 반드시 "투구"로 시작한다
- 매 PA 타임라인의 앞 ~30%를 투구 페이즈로 고정: 마운드 위 투수(아틀라스에 pitch_set→kick→stride→cock→release 7프레임 **이미 있음**) 와인드업 → 릴리스 → **공이 마운드→홈으로 날아가는 궤적**(최소 0.35s, x4에서도 최소 6프레임 보장) → 포수 미트/타격 컨택.
- 삼진/볼넷은 투구 2~3구 요약 연출(마지막 공만 실제 궤적, 앞은 카운트 핍 점멸)로 리듬 유지.
- 수용: 아무 타석이나 420ms 버스트 캡처 시 투수 모션과 공이 최소 2프레임 이상 잡힘.

### PF-2. 공 가시성 보장 (배속 불변식)
- 타구 비행: 내야 타구 최소 0.5s, 외야/홈런 최소 0.9s 화면 체류. **배속을 올려도 공 관련 구간은 실시간 최소치 유지**(그 외 구간을 압축해 총 시간을 맞춤 — 기존 leverage 템포 리매핑 구조 재사용).
- 공 크기 1.5~2배 + 트레일 강화(이미 트레일 코드 있음 — 노출 시간이 문제).
- 수용: x4에서도 홈런 타구가 눈으로 따라가짐(버스트 캡처 2프레임+).

### PF-3. 결과는 "연기 후 배지" — 배지 선행 금지
- 아웃: 담당 수비수가 타구 지점으로 이동→포구→1루 송구→1루수 포구까지 그린 **후에** 결과 표기. (수비 루트 코드가 있다면 노출 시간/트리거 점검 — 현재 체감 0)
- 홈런: 타구 담장 통과 → 주자 베이스 러닝(최소 1.2s, 라운딩 곡선) → 홈인 → 그때 "홈런!" 버스트+점수 반영. 지금은 점수가 연출보다 먼저 바뀐다.
- 안타/볼넷: 타자 1루 이동(볼넷은 walk 포즈 천천히), 기존 주자 진루 이동 필수.
- 수용: 버스트 캡처에서 "움직이는 수비수/주자"가 아웃·안타·홈런 각각 2프레임 이상.

### PF-4. 중간 뷰포트(800~1100px) 레이아웃 붕괴
- 920px에서 필드 하단이 홈플레이트에서 잘리고(타자 발이 캔버스 경계에 걸침) 스크린 컨테이너 하단 절반이 빈 공백. 1280px·375px은 정상 — fractional/정수 스케일 전환 구간에서 **stage 높이가 canvas 실제 높이와 불일치**하는 것으로 보임. `calculatePhaserMetrics`의 cssH와 stage height 동기화, `aspect-ratio` 기반 CSS로 재검.
- 수용: 800/920/1024/1100/1280px 각각에서 필드 전체(홈플레이트 포함) 표시 + 빈 공백 없음(verify:browser 뷰포트 스윕 추가).

### PF-5. 투수 부재 근본 수리 + 라벨 앵커
- QA 2차의 B-2(수비 매핑)가 수리 커밋 후에도 **마운드 투수 부재로 재발/잔존** — 정적 수비 그리기에서 P 슬롯이 어디로 가는지 재추적(이번엔 "투구 중 P 숨김" 조건이 상시 참인지부터: `pitchingNow` 판정이 progress 전 구간에서 true로 남는 의심).
- 이름 라벨 앵커: 타자 라벨=홈플레이트 위 고정, 주자 라벨=이동 스프라이트 추적, 수비 처리자 라벨=해당 수비수 위. 지금은 내야 중앙에 부유(f08).

### PF-6. 홈런 카메라 팔로우 (Q-4 최소분)
- 공이 안 보이는 문제의 보완: 홈런 타구만이라도 카메라가 공을 따라 외야로 팬(줌 ≤12%, easeInOut, reduced-motion 시 오프). 일반 타석은 고정 유지.

## 3. 가드레일

1. 총 재생 시간은 현행 ±15% 이내(P 시리즈 배속/홀드/스텝과 충돌 금지). 공 가시성 최소치는 구간 재배분으로.
2. 경기 결과 무변경, `hashParts` 결정론, verify 3종 PASS.
3. 게임 결과 반영(점수/카운트 HUD)은 해당 연기가 끝나는 progress 지점과 동기화(PF-3).

## 4. 검증 (이번 지시의 핵심 습관)

- `tools/capture_gamecast_shots.mjs`에 **버스트 모드** 추가: `node tools/capture_gamecast_shots.mjs burst` → x1에서 420ms×16프레임 저장. 수리 후 체크리스트: ①투수 와인드업 보임 ②투구 공 보임 ③타구 보임 ④수비/주자 움직임 보임 ⑤배지가 연기 뒤에 뜸.
- verify:browser에 자동 체크: PA 재생 중 캔버스 픽셀 샘플로 "공 색(#fffefb) 픽셀이 마운드-홈 경로에 등장하는 프레임 존재" 확인.
- 뷰포트 스윕(800/920/1100) 클리핑 0 검사.

```powershell
npm run verify && npm run verify:browser && node tools/capture_gamecast_shots.mjs burst
```
