# 저장/불러오기 시스템 메모

## 방향

현재 저장 시스템은 `localStorage`를 사용하지 않고 JSON 문자열을 주고받는 순수 함수로만 구성한다. `src/save.js`는 DOM, 파일 입력, Blob 다운로드, 브라우저 저장소에 접근하지 않으며, UI 계층이 나중에 다운로드 버튼이나 파일 선택기를 붙일 때 재사용할 수 있는 핵심 로직만 담당한다.

## localStorage를 쓰지 않는 이유

- 웹 버전과 데스크톱 버전이 같은 origin이나 개발 서버를 공유할 때 이전 테스트 저장값이 남아 새 실행을 오염시킬 수 있다.
- 자동 저장이 아직 없는 단계에서는 저장 시점과 불러오기 시점이 명확한 JSON 파일 방식이 디버깅하기 쉽다.
- `localStorage`는 문자열 저장소라 구조 검증 없이 읽으면 오래된 상태, 깨진 상태, 다른 빌드의 상태가 조용히 섞일 위험이 있다.
- 사용자에게 파일로 내보내기/가져오기를 제공하면 저장본을 백업하거나 공유하기 쉽고, 문제 재현용 fixture로도 사용할 수 있다.

## JSON 저장의 역할

JSON 저장은 현재 프로토타입에 맞는 가벼운 영속화 방식이다. `exportGameState(state)`는 검증된 상태를 안정적인 저장 포맷으로 직렬화하고, `importGameState(jsonText)`는 JSON 파싱 후 팀 배열과 선수 배열의 기본 구조를 확인한 뒤 상태 객체를 반환한다. 잘못된 팀/선수 구조는 명확한 에러로 막아서 시뮬레이션 엔진이나 UI까지 오염된 데이터가 흘러가지 않게 한다.

## 나중에 SQLite로 가는 이유

시즌 기록, 선수별 누적 기록, 트레이드 이력, 저장 슬롯, 스카우팅 리포트처럼 데이터가 커지고 관계가 늘어나면 단일 JSON 파일은 검색과 부분 업데이트가 불편해진다. 그 단계에서는 SQLite가 더 적합하다.

- 저장 슬롯과 시즌 로그를 테이블로 나누어 관리할 수 있다.
- 선수/팀/경기 기록을 인덱싱해 빠르게 조회할 수 있다.
- 데스크톱 앱에서는 로컬 파일 기반 DB로 배포와 백업이 단순하다.
- JSON import/export는 SQLite 도입 이후에도 백업, 호환성 마이그레이션, 버그 리포트 첨부용으로 유지할 수 있다.

## 현재 export 함수

- `exportGameState(state)`: 상태 구조를 검증하고 JSON 저장 문자열을 만든다.
- `importGameState(jsonText)`: JSON 문자열을 파싱하고 구조 검증을 거친 상태 객체를 반환한다.
- `makeSaveFileName(state)`: 현재 날짜, 일차, 선택 팀을 바탕으로 저장 파일명을 만든다.
- `validateGameState(candidate)`: 팀 배열과 각 팀의 선수 배열 기본 구조를 검증하고 JSON 안전 복제본을 반환한다.
